# 🚀 NestJS Microservices Monorepo

Бэкенд-приложение построенное на **NestJS** по паттерну микросервисной архитектуры.  
Монорепозиторий включает **7 микросервисов**, взаимодействующих через gRPC, REST, Kafka и RabbitMQ.

---

## 🧩 Стек технологий

| Категория          | Технологии                             |
| ------------------ | -------------------------------------- |
| **Транспорт**      | gRPC · REST (proxy) · Kafka · RabbitMQ |
| **БД и кэш**       | PostgreSQL (Prisma) · MongoDB · Redis  |
| **Хранилище**      | MinIO (S3-compatible)                  |
| **Очереди**        | BullJS                                 |
| **Инфраструктура** | Docker Compose                         |

---

## 🗂 Микросервисы

### 🔀 Gateway

> Точка входа. Принимает все запросы, валидирует DTO и маршрутизирует их по gRPC в нужные микросервисы.

- JWT-аутентификация через декоратор `@Protected` (JWT-стратегия + Role Guard)
- Глобальный **Throttler** (rate limiter)
- Глобальный логгер входящих запросов + измерение времени ответов
- DTO-валидация через Pipes
- Кастомные декораторы: `@Id`, `@RefreshToken`, `@IdempotencyKey`
- Swagger-документация по адресу `/docs`
- Filter gRPC ошибок

---

### 👤 User Service

> Управление пользователями: регистрация, аутентификация, профили, балансы, сессии.

- Refresh-токен в сессии и куки
- Хэширование паролей через **argon2** с pepper
- Soft-delete пользователей
- OTP-коды с кулдауном, лимитом попыток и TTL — хранятся в **Redis**
- Пагинация и фильтрация списка пользователей
- Кэширование в **Redis** (бинарные данные) с инвалидацией через EventEmitter
- Оптимизированный поиск активных пользователей с использованием индексов и анализом через `EXPLAIN ANALYZE`

---

### 📁 File Service

> S3-совместимое хранилище аватарок на базе MinIO.

- Хранение аватарок в **MinIO**
- Загрузка через стримы
- Валидация типа и размера файла
- Проксирование файлов напрямую из gateway (без загрузки в память)
- Soft-delete аватарок

---

### ⏱ Job Service

> Выполнение фоновых и крон-задач.

- Крон-задача на сброс балансов всех пользователей каждые 10 минут
- Реализована через очередь **BullJS**

---

### 💸 Transaction Service

> Пополнение и перевод средств между пользователями.

- Полная история операций через сущность `Transaction`
- Идемпотентность операций
- Пессимистичные блокировки для предотвращения гонок данных
- Предотвращение дедлоков через фиксированный порядок блокировок
- Учёт уровней изоляции и транзакционных аномалий
- Хореографическая сага между `transaction-service` и `user-service` для атомарного обновления балансов + Outbox Event
- Интеграция с тестовой платёжной системой **Yookassa** (webhook)

---

### 📧 Mail Service

> Отправка OTP-кодов и уведомлений по email.

- Получает сообщения через **RabbitMQ**
- Отправка писем через **SMTP** в локальный почтовый ящик **Mailpit**

---

### 🔔 Notification Service

> WebSocket-сервер для реалтайм-уведомлений пользователей.

- WSS-сервер с валидацией JWT при подключении
- Слежение за сроком жизни токена через LBV-таймеры
- Получение событий о транзакциях через **RabbitMQ**
- Доставка уведомлений пользователю по **WebSocket**
- Хранение истории уведомлений в **MongoDB**

## 📦 Компоненты монорепозитория

```
monorepo/
├── contracts/       # Контракты и интерфейсы для межсервисного общения
│                    # (gRPC-proto, proxy-headers, RMQ/Kafka payloads, брокерные события)
│
├── registries/      # Реестры gRPC-клиентов, консьюмеров и очередей
│                    # Упрощённая интеграция через кастомные библиотеки
│
├── libs/            # Кастомные библиотеки: утилиты, абстракции, билдеры и фабрики
│                    # Инкапсуляция логики интеграции межсервисных и инфраструктурных клиентов
│
└── shared/          # NestJS-независимые сущности: enums, interfaces, extensions
```

---

## 🏗 Архитектура каждого сервиса

Все сервисы придерживаются единого архитектурного стиля:

```
src/
├── common/      # Декораторы, фильтры, guards, interceptors, middleware, pipes, стратегии
├── config/      # Валидация ENV через Zod, конфигурационный модуль
├── infra/       # Инфраструктура: Redis, Prisma, MinIO (S3), MongoDB
├── core/        # Переиспользуемые модули: hash, otp, session, throttler,
│                # logger, cron, events, files, queue и др.
├── feat/        # Доменные фичи и бизнес логика: auth, account, avatar, balance, user, profile
├── processes/   # Фоновые процессы: Kafka/RMQ контроллеры, WebSocket, хуки
└── integrate/   # Интеграции с внешними сервисами (Yookassa и др.)
```

---

## 🛡 Дополнительно

- Валидация ENV-переменных через **Zod** в каждом сервисе
- Межсервисные guards с настройкой access token list
- Pre-commit хуки через **Husky**
- Единый архитектурный стиль во всех микросервисах

```
User Service
  ├── gRPC ──────────────► Job Service
  ├── gRPC ──────────────► Transaction Service
  ├── gRPC ──────────────► File Service
  ├── Kafka saga ─────────► Transaction Service
  ├── RMQ ───────────────► Mail Service
  └── RMQ ───────────────► Notification Service
```
